Tutorial: Designing a GraphQL API

Step1. A Bird’s-Eye View

設計 Schema 時，永遠先以高層次物件的結構與關係為方向去思考，以 ER model 概念大略設計schema後，再去處理細節的欄位。在 Query 裡建議使用「名詞」而非「動名詞」，更能夠表達 Graph 的概念。

Step2. A Clean Slate

考量API實作細節：假設 Author 與 Post 是多對多的關係。
以 ER model 的概念來說，必須儲存 Author 與 Post 多對多的關聯資料，但這並不是 API 實現的部分，故剔除多對多的 schema。

API設計上不暴露實作細節：以案例來說，如果 Post 分為教師公告與學生發問 2種，合併為同一個 Post schema 隱藏實作細節，是較好的設計，但因情況不同可自行斟酌。

依照商務領域去設計 schema，而不應完全實作 GUI 所需的欄位或是複製傳統的 Restful API。例如教師公告與學生發問是 2 種不同 GUI，但仍可歸類為同一種 schema。

Step3. Adding Detail

在完成上述作業後，我們可以重新加上所需的欄位。要注意 GraphQL 是公開文件，應由實際需求和案例決定，不要隨意透露資料庫欄位細節。
新增欄位簡單，但是更改或刪除它們難度不低。尤其是標記 ! 的必填欄位，優點是可以強化 Schema 的驗證邏輯，但缺點也不少，比如

  1. 難以擴展 Schema
    修改一個 non-null 欄位意味著 breaking change 。預測未來的 Schema 變化是不實際的，把 nullable 欄位轉向 non-null 很容易，但要轉回來的成本會高出很多。
    打個比方，有可能今天有個 User type ，規定一定要有名字與生日，那假如有一天註冊方式改用 Oauth 或其他方式使得 name 非必要，或是要修改能看到 age 的權限等等，都會限制你的 Schema 。
    而且因為前端會預期得到 non-null ，所以如果 deprecate 掉這個欄位，還是要提供 default 值。
  2. Non-null 欄位可能會放大錯誤

比較適合用 ! 標記必填欄位的地方

  1. id 欄位
  2. parent Object
  3. Boolean type: boolean 欄位有 null 值是不應該存在的
  4. [type!] 欄位: 一個有值為 null 的 array 是不應該存在的

在大型的 GraphQL Schema 中，一般會推薦所有主要商業邏輯物件都要實作 Node interface type，因為通常這些物件在 database 中都有 id ，實作 Node interface type 可以明確告訴 Client 這是一個重要概念的物件，並且可透過 id 的操作來做 caching 及 batching。

密切相關的欄位可以單獨抽離出為子 type。
例如 Post的 title 和 author_name 這 2 個欄位關係密切，但可能必須考慮另外一個狀況：如果 Post 為自動產生，則 Author 可能為空值。所以將這 2 個欄位獨立為 1 個 type，且 Post 的 Author 不設為 Not null 是較好的設計。

隨時確認資料是否做分頁。

  • offset-based

    • 優點：可直接跳到指定分頁
    • 缺點：資料量越大，offset消耗效能越大。資料異動時，分頁起始資料可能錯誤。
    • 適合資料不常異動、效能要求不高及需要顯示分頁的功能（如部分後台管理功能）

  • cursor-based

    • 優點：資料量大時，仍有不錯效能。資料異動時，不會發生錯誤。
    • 缺點：無法跳至指定分頁。需另外為 cursor 欄位建立 index。
    • 適合資料大量異動、效能要求較高及資料為漸進式呈現。(如下捲式分頁)

應該將物件關聯用的ID欄位(類似外來鍵FK欄位) 獨立出 type，取代原有的 ID 欄位。
在 Restful API 的設計上，我們很常發生一個物件有多個關聯用 ID 欄位，藉這些 ID 欄位來取得其他物件的相關資料。但在 GraphQL，這是一個 anti-pattern，這是為了確保每一個 type 物件只會有一個 id 作為辨識。
若獨立的出來的 type 該 id 欄位早已在其他 type 存在，那事實上可以考慮用既有 type 來取代即可，利用 FK 資料串接可於 model 和 resolver 實作

取一個比較平易近人而非過於技術的名稱
例如: bodyHtml 改名為 description

GraphQL 預設總共有 5 種 Scalar Type。因 GraphQL 的一大強處就是 type validation ，但當功能需求越開越多時，開始會存取 url, date, positive integer 等等特定格式的資料，只使用 String 或 Int 做 type validation 的效益不高，過於寬鬆。這時候可以自定義的 Scalar Type 來幫助你達到以下三點：

  1. 讓 Clinet 與 Server 的開發者對於資料格式有共同的認知
  2. 強迫 Client 的送出正確格式的 query
  3. 強迫 Server 的回覆正確格式的 response

Scalar Type 擴充包
https://github.com/Urigo/graphql-scalars

那麼什麼時候我們需要 Custom Scalar Type?

  1. 當你需要一個有特殊語意的值時，就使用 Custom Scalar Type
  2. 但如果你的值是比較模糊的格式如 email，這類在 client 端檢查較為複雜的格式的話，建議使用寬一點的 type 如 String
  3. 而如果你的值是定義比較明確如 DateTime 就是要 ISO 格式不接受其他，那就可以考慮使用 Custom Scalar Type 來規範 Client 不能亂傳

開發初期可先使用 String，只有當對於特殊語意的 Scalar Type 的需求夠多時，比如每個 Business Model 都要有個 createdAt ，那可以考慮使用為 createdAt 建立一個 Date Scalar Type 。

否則有時候 Custom Scalar Type 也是把雙面刃，用錯時機點可能造成 Schema 規範過多不夠彈性。

最後務求團隊對於所有 Scalar Type 使用上是有共識的。

在 schema 使用 enum 來定義值集合。

Step4. Business Logic

API 應該不只提供資料，也要能呈現商業邏輯。複雜的計算應該交由後端來處理，而非散落各處。
例如有一個 Author 底下有 posts 清單欄位，但前端這時可能只需要書本數量，這時可以提供一個 postCount 欄位，讓前端不用取得 posts 清單來計算數量。

即時呈現了商業邏輯，也還是要提供 raw data。前端仍有可能想要自己處理資料邏輯。

Step5. Mutation

一個邏輯行為寫一個 mutation。
例如：一個商品如果所有狀態更新寫在 update，會讓狀況顯得混亂，故先拆成基本的 5 個 mutation

  • create
  • update
  • delete
  • publish
  • unpublish

我們仍然覺得部分 mutation 負擔太多責任，並試圖去解構它。大略上有以下 3 種常見方式

  1. 一次更新所有資料。例如：updateOrderProducts (products: [Product!]!)
  2. 一次更新部分資料。例如：deleteOrderProducts(ids: [ID!]!)
  3. 一次更新一筆資料。例如：addProduct(input: Product!)

經驗上來說，可以考量以下幾點作為選擇依據：

  1. 當資料量大、適合第二三種，但如果資料量不大且結構簡單，那第一種最簡單。
  2. 一次做大量簡單操作時，第一三種皆可，第二種比較難用 ids 來表達操作內容。
  3. 如果一次更新父物件與子物件時：
  • 只是修改關係 (如 user 與 collection)，這三種方式皆可
  • 有強烈關聯或修改資料時 (如 user 與 order)，這建議第三種方式

如果是處理物件關係資料的 mutation，可以考慮批次處理
最後 mutation 會拆成如下

  • create
  • update
  • delete
  • publish
  • unpublish
  • addProducts
  • removeProducts
  • reorderProducts

Input: Structure, Part 1

在看完 Mutation，我們來看 input 的部分。在看過許多真實環境資料，常見到有人會為多個 mutation 定義一個共同的 input 去處理所有的屬性，這並不是推薦的設計方式，我們應盡可能讓 input 簡單。

  1. 針對許多簡單的 mutation，我們只需要用 ID/IDs 來處理，例如：
  • delete, publish and unpublish 使用 collection ID
  • addProducts, removeProducts 使用 collection IDs 和 product IDs
  1. 針對較為複雜的 mutation，則需要 input 來支援，例如：
  • create
  • update
  • reorderProducts

Shopify 內部制定的規則為 TypeName，例如 orderCreate, orderUpdate, orderRemove，在 GraphQL API排序上較為整齊。不過這基本上見仁見智，只要團隊規範好命名方式，然後持續遵守就可以了。

Input: Scalars

針對 input field，除了語意上必定要做處理的欄位，否則不隨意加上必填屬性，因為你無法預期客戶的需求

輸入的格式若是非常明確但前端驗證相對複雜，就建議用 Weaker Type ，讓後端去跑驗證程序。比如要輸入 Html 字串，就建議直接用 String ，接下來交給後端去檢查，一方面後端可以做更仔細的檢查，另一方面也避免前後端各自維護檢查邏輯的程式。

如果輸入的格式模糊 (可能值很多)且前端驗證相對簡單，但就很適合使用 Strong Type (也就是使用 custom scalar type) 。比如 Date 格式，前端可能輸入千百種格式 (Unix timestamp、ISO、日期字串…) ，這時候限制前端輸入的 Date 格式，而前端也只需要做個日期選擇器，就避免了很多不必要的驗證與檢查。
另外一般欄位也非常推薦使用 enum 這種 strong type scalar。

Output

在設計 mutation output type 時，最簡單的就是直接回傳被更改的 type ，如 authorUpdate 就應該回傳 Author type 更新後的資料，但軟體的設計常常跟不上需求，有可能今天需要增加錯誤處理邏輯或是提供更多 mutation 的詳細資訊如成功/失敗次數時，只用一個現成的 object type 就有了擴展的困難。

因此，可以考慮為每一個 mutation 回傳資料都設計一個專屬的 object type，authorUpdate 的 Output 為 AuthorUpdatePayload

這邊 shopify 建議可以定義一個 userErrors 來表達user/business-level 錯誤，而那種 data 層級的 error 交給 GraphQL 的 Type Validation。這邊 [UserError!]! 保證就算沒有錯誤也會回傳空 array。

    • Migration 的策略：採用 Incremental adoption，分為五個階段。每個階段都是shippable 且 fully funcational 的產品。
      • 第一階段：Restful → GraphQL。實作 Adapater 過渡、根據 Gql Schema 用 codegen 自動產出對應的 typescript interface。
      • 第二階段：Propagate Types。加強 TypeScript Types，來提高後續階段的實作信心。在此階段不碰觸任何 Runtime behavior 與產品行為。

Chapter2 重構的原理 - 整理筆記

Posted on 2020-02-19

Chapter2 重構的原理 - 整理筆記

何謂重構?

定義

  • 不改變行為、Input、Output 的情況下，用一系列的重構手法對軟體結構進行變動

效果

  • 提高程式的可讀性
  • 減低未來修改程式或新增功能時所耗費的成本

簡述

  • 一個重構的過程應該是一連串的小步驟，一個小步驟短則５分鐘，最長應於數小時內解決。
  • 性能特徵或許會有變化，但應讓使用者感受不到變化
  • 原本就有 bug 的話，重構後仍應繼續存在。在確認重構完成後，再開始修正。
  • 重構 vs 性能優化：兩者都是在不改變行為、Input、Output 的情況下進行優化。但是方向不同，重構後有可能導致性能下降，性能優化後也可能導致程式可讀性下降。應盡量在當中取得一個平衡，也就是【讓使用者感受不到變化】的前提下進行重構。

兩頂帽子：加入功能與重構

  • 兩者不會同時進行，而是一個不斷輪流進行的過程
  • 重構 → 加入功能 → 加入測試 → 重構 → …

為何重構

重構可以改善軟體的設計

  • 程式架構設計隨著需求修改，逐漸劣化，讓可讀性越來越差。
  • 消除重複的程式碼：確保每一段程式碼只描述任何事項一次，修改程式碼只需修改一個地方

重構可讓軟體更容易了解、更容易找出 bug

  • 寫出容易閱讀的程式碼，減少修改難度及修改時間，幫助自己也是幫助別人。
  • 程式碼容易閱讀後，更容易釐清思路看出 bug

重構可以協助提升程式編寫速度

  • 刻版印象：重構是花時間且沒有生產力的
  • 重構後通常能有效減低未來修改程式或新增功能時所耗費的時間成本
  • 我們很難一開始就做出良好的設計，初期一定會需要重構來打造更好的基礎。因此從長遠來看，為了未來快速完成新功能，重構非常的重要。

何時該進行重構

同樣的事做3次，就該重構了

重構的種類

  • 伺機性重構：不一定要預留時間重構，通常在加功能或除錯時，自然而然的調整

    • 預備性重構：便於加入新功能或擴充現有函式。Parameterize Function (358)
    • 理解性重構：讓程式碼更容易被了解。可藉由這過程釐清自己對於程式的了解。
    • 打掃性重構：程式碼重複或有一些不必要的邏輯需要做整理。此類重構優先度不高，有空再處理。

  • 計畫性重構：優秀的程式碼也需要重構，以保持更好的狀態迎接新功能。

  • 長期重構：分階段逐步完成大規模重構。例如 Restful API 轉為 GraphQL

  • 程式碼複審時重構：讓團隊對於重構有共識，傳達更好的經驗給團隊，提升整個團隊能力。近似 pair programming

不要說出來(?)

何時不該重構

  • 程式過於混亂，但不需修改的時候
  • 重寫比重構簡單的時候
  • 加一個小的新功能，需要大規模的重構，效益不成正比

重構的難題

減緩新功能的完成速度？

重構的目的不是為了展示程式碼有多出色，它完全是出於經濟目的。
重構提升我們的開發速度、更快地添加新功能、更快地修復 Bug。

程式碼擁有權

你可能因為程式碼是別的部門管理，或是程式碼是提供給顧客使用的API，難以對程式碼進行重構
這時可以考慮使用 Rename Function (147)作為一個中介，不過這是沒有辦法中的辦法
最理想的狀態還是每個人都程式碼擁有權。

分支

分支存活的時間越久，把重構成果合併至主線的難度越高
作者建議每天至少要和主線合併一次

測試

  • 用自檢(self-testing)程式碼確保每次重構後的行為都維持不變
  • 信任優秀的自動重構功能：例如 VScode 的重新命名符號

老舊的程式碼

  • 在程式中尋找可以插入測試的接縫，但這仍需承擔一定程度的風險
  • 不建議一次重構完畢，試著分解成多個小部分，每次都改善一小部分，逐步改善程式碼體質

資料庫

預先撰寫遷移腳本 migration script 來執行 table schema 的重要修改，如：欄位名稱重新命名。

重構、架構與Yangi

  • 軟體架構設計往往被認為一開始就要一次到位，事實上是需要根據需求不斷的重構來優化軟體架構。
  • 為了不一定會發生的未來增加架構的彈性，事實上是一種過度設計，反而會降低對變化的應變能力。
  • 除非認為未來重構這塊比較困難，否則不需現在加入彈性機制，只要就現行的需求做到適當的重構即可。

重構與更廣泛的軟體開發程序

  • 持續整合、自檢程式碼以及重構 (test-drivern development)
  • 快速回應、小步修改、持續交付，讓軟體持續處於可發表的狀態

重構與性能

  • 重構常常被認為會為了可讀性而寫出降低性能的程式碼。
  • 作者認為，編寫快速軟體的訣竅，應是先重構出一個可讀性高、可調整的軟體，再調整成滿意的速度，在可讀性與性能間取得平衡。
  • 性能優化應著重於 bottleneck (後端通常是資料庫連線與 SQL 查詢)

如何同步2個不同Database Name的Table Schema (MySQL)

Posted on 2018-12-26 Edited on 2019-10-29

前言

同步跨環境的 table schema 一直是一件繁雜卻又非常重要的任務之一。市面上同步不同 Database 的 Table Schema 工具不勝枚舉，甚至能自動化，但能同時處理以下 2 種情況的工具並不多見。

  1. 來源 Database 與目的 Database 的 Database Name 不一致。
  2. 來源 Database 與目的 Database 不在同一個 VPC。

通常正式環境的 Database 都會透過 VPC 保護。如果為了能夠同步 Table Schema，而在 VPC 開一個洞 (e.g. VPC peering)讓工具連進VPC，難免心裡毛毛的。因此本篇文章介紹一個「怪招」來解決以上 2 個問題。

解法

這邊會利用到 MySQL 官方提供工具 MySQL Workbench。

  1. 利用 MySQL Workbench 建立來源端 Database ER model

  1. 將來源 Database 的 Schema Rename 為目的 Database 的 Database Name，並存檔。

  1. 連進目的 Database 所在 VPC，並開啟 Rename 後的 ER model 檔，最後選擇 synchronize with Any Source。

  1. 最後因 ER model 的偽裝成功，讓兩個不同 Database Name 的 Database 得以進行比對，並產生建立或修改 table schema 的語法。

結論

這個方法其實沒有太大的學問，卻能解決市面上許多比對工具所遭遇到的限制。當然，我相信一定有更好、更方便的工具來簡化這流程，也歡迎各位推薦。

AWS Lambda Rest API：利用 serverless framework從開發到部屬

Posted on 2018-08-13 Edited on 2019-10-29

前言

相信大家對於 serverless 並不陌生，所以在此並不會討論 serverless 與傳統專案的架構差異與優缺點。這次分享的是我使用 Node.js 8.10 開發 lambda function，利用 serverless framework 幫我進行部屬，在這個開發過程中注意的要點與踩到的地雷 😬。

Why serverless

當中我們也參考了 Terraform, Apex 等知名部屬工具。Terraform 的強項是偏重於雲端基礎架構管理，Apex 則是專精於 AWS lambda function 的打包與部署，對於其他 AWS 資源的連動與設定較為缺乏。相較之下，serverless 除了著重於 function 的開發外，也支援 API Gateway, Cloudwatch Schedule 的連動設定，開發社群也提供許多外掛加強 serverless 的特異功能。更重要的是不只支援 AWS，同時也支援 Azure, GCP, IBM OpenWhisk，以防有個萬一 👻。

About Lambda function

Lambda function 是基於容器技術，以事件觸發條件而啟動，可由HTTP、API呼叫、AWS或第三方服務來啟動。本篇文章是以串接 API Gateway，API 呼叫的方式來啟動為例。在啟動 Lambda function 前，有幾項參數是必須了解與設定。

  1. Run time：執行環境，目前有 C#, Java, Python, Go, Node.js 可選擇

  2. 處理常式：程式開始執行點

  3. 環境變數：視環境與需求設定，常搭配 AWS KMS 加密後字串使用

  4. 執行角色：於AWS IAM設定，如需賦予 function 存取其他 AWS 服務的權限可在此設定

  5. 基本設定：設定 function 記憶體與逾時時間，AWS 根據記憶體設定與執行時間收錢。記憶體設定越高，所綁定的硬體資源越強大。

  6. 網路：如有使用 VPC 或 Security Group 進行存取控管，可於此設定。

  7. DLQ資源：如錯誤發生後超過 retry 次數，可設定是否連動 SNS, SQS。

  8. 並行數量：如多人同時執行 lambda function，可限定最多多少人同時執行。通常用於有執行資料庫查詢的狀況，以避免資料庫 connection 用光。

About Lambda Project

在 FaaS 的概念中，每一個 function 都是獨立的 Service。聽起來是一個很美好的概念，但在實務上，每開發一個 lambda function 就必須打包再上傳，其實是一件很累人的事情，更遑論一個 CURD 可能就有 4 個 function。因此我們有必要去建立一個 Project 集中管理 Lambda function，以利進行版控及集中部屬。

那我們就從安裝 serverless 開始吧！

npm install serverless --save

因為專案的性質，開發者的喜好，專案的架構可能會有所不同，但只要是利用 serverless 進行開發與部屬，必定會有一個 serverless.yml 檔作為 serverless 部屬時的參考依據。現在我們假定需求是這樣

  1. 建立 2 支 API：新增使用者、取得使用者資訊。Route分別為
    POST /v1/user
    GET /v1/user/:userUid

  2. 區分 dev 與 prd 環境。

到這邊你可能會發現幾個陌生的屬性，這邊特別說明如下：

  1. plugins:
    前面提到，serverless 開發社群提供相當多的套件，在這邊我們使用到的 serverless-domain-manager，主要是協助我們部屬時，將 lambda function 串接 API ，而 API Gateway 相關設定則設定於 customDomain 屬性當中。

  2. deploymentBucket:
    前面提到，serverless 利用 AWS cloudformation 幫我們進行部屬時，過程中會建立一個亂數名稱的 S3 bucket。為避免每次部署就建一個 S3 bucket，產生許多無用的 bucket，在這邊強制指定部屬時的中繼 S3 bucket。

  3. TZ:
    lambda 預設時區為 UTC。如有指定時區需求，可在此進行設定。

Minimize your lambda function

在不斷進行開發與部屬一段時間後，你可能會發現一件事：每個 lambda function 檔案大小都一樣大！Lambda function 是基於容器技術啟動，這也意味著檔案大小會影響啟動 function 的速度。假設 animal function 只使用 cat 套件，卻因為 people function 使用了 time 套件，使得 animal function 被迫加入了 time 套件，會無謂的增加 animal function 的檔案大小。

這時我們可以使用一個外掛 serverless-plugin-include-dependencies，來幫助我們每個 function 在部屬時，只抓取有用到的套件進行打包部屬，以減少 function 檔案大小。開發時也應養成好習慣，如果自己短時間能開發完成的小功能，便不應該依賴套件。

npm install serverless-plugin-include-dependencies --save-dev

並於serverless.yml 進行相關設定

plugins:
  - serverless-plugin-include-dependencies

Warmup lambda function

好不容易完成並部屬了 API ，這時前端跟你反應 response time 有點慢！ 😫這其實是 Lambda 使用容器技術的一個硬傷：需要短暫的啟動時間。只要該 function 長時間未被執行， Lambda 容器就會被刪除，導致下次執行必須花費額外的啟動時間。這時我們該怎麼辦呢？

沒錯，就是要定期去執行它了 😢

這時我們可以利用一個外掛 serverless-plugin-warmup，這外掛原理非常簡單：就是收集需要定期執行的 lambda function，利用 cloudwatch schedule 進行排程呼叫 lambda function。一般建議是設定 10~15分鐘，就可以避免 lambda 容器被 AWS 刪除。另外別忘記幫每個 Lambda function 針對此外掛的呼叫直接在一開始就 response，以避免消耗資料庫與 Lambda 計算資源。

npm install serverless-plugin-warmup --save-dev

並於serverless.yml 進行相關設定

plugins:
  - serverless-plugin-warmup
custom:
  warmup:
    folderName: '_warmup'
    cleanFolder: false
    memorySize: 128
    name: 'warmup'
    schedule: 'cron(0/10 0-16 ? * MON-SUN *)'
    timeout: 20
prewarm: true

functions:
  add-user:
    warmup: true

承上範例，如果我們要針對 get-user 初始化一個單元測試的檔案，可以使用下列指令

serverless create test -f get-user

撰寫好你的測試案例後，可以使用下列指令來測試你的案例

serverless invoke test --s dev -f get-user

Reaching CloudFormation resources limit

隨著你寫的 function 越來越多，最後部屬上一定會遇到一個問題：Reaching CloudFormation resources limit。發生這問題的原因在於，serverless 是利用Cloudformation 進行一連串的部屬，然而一次能部屬的資源是有上限的（200個）。這時有 3 個 plug-in 可以解決問題，詳情參閱此文章。

Integrated with CI/CD

在完成這一系列的 serverless REST API project 建置後，要串接 CI/CD 也是沒有問題的。只要 commit 到 git 中，再交由 CI/CD 監聽事件，自動執行 unit test 與 deploy 指令，即可完成自動部屬。

結語

在這次的 AWS Lambda Rest API 的建置過程當中，確實感受到 serverless 的魅力，高可用性、低耦合性、價格便宜，很適合亟需成長的新創公司快速建立服務，但也發現在很多應用上，仍有其軟硬體限制，綁手綁腳。但說穿了，對後端工程師來說，就是多一項武器可以使用，針對各種不同的環境與狀況，決定適用的專案架構，才是正確面對 serverless 的態度。
